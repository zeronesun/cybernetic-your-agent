# Глава 5: Практическое внедрение и развертывание

В этой главе мы покажем, как развернуть эту архитектуру на вашей платформе агентов. Мы продемонстрируем конкретные команды и файлы для подготовки, настройки и проверки.

## 5.1 Подготовка и настройка

### 5.1.1 Обязательные файлы

Вам нужно создать/подготовить следующие файлы:

| Файл | Расположение | Назначение |
| :--- | :--- | :--- |
| **L1-метакогнитивный фреймворк** | `skills/cybernetics-feedforward-startup.md` | Правило приёма задач и交付自检 |
| **Таблица управления баллами** | `memory.md` (с записью `积分控制表`) | Отслеживание реакций и накопления |
| **L3-паттерны сцены** | `memory.md` (с записями `L3*`) | Шаблоны для повторяющихся задач |
| **Навыки (Skills)** | `skills/` (разделённые файлы `SKILL.md`) | Процедурные знания, пошаговые инструкции |

### 5.1.2 Создание L1-метакогнитивного фреймворка

Создайте файл `skills/cybernetics-feedforward-startup.md`:

```markdown
---
name: cybernetics-feedforward-startup
category: feedback-loop-monitoring
---

# Cybernetics Feedforward Startup — запуск с предзагрузкой

## When to use
- При запуске новой задачи (перед началом работы)
- Перед выполнением любого действия

## Цель
Выполнить механизм предзагрузки (feedforward), чтобы загрузить пользовательские предпочтения, исторические уроки и шаблоны избегания ошибок.

## Процесс

### Шаг 1: Загрузить пользовательские предпочтения
1. Поиск в памяти: `prefers`、`喜欢`、`prefers`、`ценивает`、`не喜`、`不喜`
2. Отфильтровать соответствующие записи (с таймштампами)
3. Сортировать по времени, взять последние 3-5 записей
4. Вставить в контекст как `[USER PREFERENCE]`

### Шаг 2: Загрузить таблицу управления баллами
1. Поиск в памяти: `积分控制表`、`积分控制`、`积分`
2. Найти саму таблицу `积分控制表` (если существует)
3. Фильтровать: только записи с `积分计数 >= 阈值 (2)`
4. Вставить в контекст как `警告: [积分控制] 该模式已触发≥2次, 必须遵循该规则.`

### Шаг 3: Загрузить L3-паттерны сцены
1. Определить тип текущей задачи
2. Поиск в памяти: `L3`、`核心pattern`、`pattern`
3. Загрузить шаблоны для этого типа задач
4. Вставить в контекст как reference

### Шаг 4: Вывести список возможных проблем
- На основе загруженных данных, сформировать список моментов, которых нужно избегать
- Включить:

```
【Frame of Reference】
- Пользовательские предпочтения: (из загрузки)
- Предупреждения таблицы управления баллами: (из таблицы)
- L3-паттерны сцены: (из шаблонов)

【Avoidance Checklist】
- (перечень типичных ошибок для этой задачи)
```

## Успешные индикаторы
- Пользовательские предпочтения загружены и учитываются
- Предупреждения из таблицы учтены в планы
- План не противоречит L3-паттернам
```

### 5.1.3 Создание навыка delivering-self-check

Создайте файл `skills/cybernetics-delivery-gate.md`:

```markdown
---
name: cybernetics-delivery-gate
category: feedback-loop-monitoring
---

# Cybernetics Delivery Gate — проверка перед выводом

## When to use
- Перед выводом любого результата пользователю
- После завершения работы над задачей, перед ответом

## Цель
Выполнить трёхуровневую проверку рисков (P0/P1/P2/P3) перед выводом. Только когда все проверки пройдены — выводить.

## Процесс

### P0: Критические проверки
- [ ] **Синтаксис**: Python/Shell синтаксис валиден
- [ ] **Ошибка выполнения**: Команда не вернула `error` и `exit_code == 0`
- [ ] **Типы**: Использованы правильные типы данных (напр., не строка вместо списка)

### P1: Проверки функциональности
- [ ] **Порядок файловых операций**: Используется безопасный порядок «Запись → Проверка → Удаление»
  - Не использован опасный порядок «Удаление → Запись»
- [ ] **Целостность данных**: Тесты покрывают все ключевые сценарии
- [ ] **Права**: Нет попыток использовать `sudo` (запрещено)

### P2: Проверки качества
- [ ] **Избыточность**: Ответ не содержит >50% повторяющихся фраз или бессмысленных шаблонов
- [ ] **Единообразие терминов**:同一个术语 не смешаны китайский/английский
- [ ] **Единообразие формата**: Один и тот же тип данных всегда представлен одинаково

### P3: Опциональные улучшения (не требуются)
- [ ] **Читаемость**: Имена переменных/функций достаточно длинные и описательные
- [ ] **Производительность**: Нет дублирующихся вычислений, которые можно кэшировать

## Успешные индикаторы
- Все P0 и P1 проверки пройдены
- Любые P2 проблемы были исправлены или учтены
- Результат выводится только после успешной проверки

## Типичные ошибки
- ❌ **Пропуск проверки**: Напрямую выводить без проверки
- ❌ **Игнорирование P0/P1**：「проходит кому как есть»
- ❌ **Не исправлять P2**: выводить с очевидными проблемами стиля
```

### 5.1.4 Создание навыка Closed-loop Review

Создайте файл `skills/cybernetics-control-loop-review.md`:

```markdown
---
name: cybernetics-control-loop-review

---

# Cybernetics Control Loop Review — углублённое закрытие цикла

## When to use
- После завершения фазы (неделя/месяц/проект, крупная поставка)
- Чемодан нельзя распаковки после завершения важного этапа
- Триггер: сказать «起飞」 (起飞/полет)

## Цель
Систематизировать, извлечь уроки и переместить знания в долговременные слои (L2/L3/Skill).

## Процесс

### Шаг 1: Просмотр таблицы управления баллами
1. Прочитать `积分控制表`
2. Проверить категории, где `累计次数 >= 阈值(2)` 且 `已下沉 = 否`
3. Для каждой такой записи:

   ```
   - Прочитать запись (类别 + 错误模式)
   - Определить соответствующий уровень (L2/L3/Skill)
   - Создать/обновить запись памяти или навык
   - Обновить запись: `已下沉 = 是`
   ```

### Шаг 2: Анализ корневых причин
Для каждого повторяющегося шаблона:

```
- Почему он повторяется?
  - Механизм загрузки пропущен?
  - Инструкция неудачна?
  - Триггер не сработал?
- Как исправить?
  - Добавитьость в L1?
  - Улучшить навык?
  - Усилитьdelivery gate check?
```

### Шаг 3: Экстракция reusable знаний
Для успешных подходов:

```
- Что приводило к успеху?
- Можно ли сделать reusable workflow?
- Преобразовать в skill?
```

### Шаг 4: Проверка целостности слоёв (层级净化)
Убедитесь, что:

- L2 содержит только принципы (Why), без пошаговых деталяций
- L3 содержит только шаблоны для типов задач, без исторического фона
- Skills содержат только процедурные знания (How), без историй пользователей

Если обнаружены нарушения, выполните correction:

```
- Вырезать каких-роверов в неправильном слое из записи
- Переместить в правильный слой
- Пересоздать entry
```

### Шаг 5: Улучшение тем (闸门升级)
Based на всеми обнаруженных проблемах, расширьте/обновите:

- **Проверки в delivery gate** (дополнительные P0/P1/P2 проверки)
- **Загрузочные триггеры в feedforward** (новые пункты доступа)
- **Action items в L1** (новые правила/процессы)

### Шаг 6: Предложение миграции (迁移提议)
Для успешных reusable workflows:

```
- Создать skill для этого workflow
- Предложить внедрить skill в другие агентов/users (как sharable asset)
- Документировать успешный паттерн
```

## Успешные индикаторы
- Все записи в `积分控制表` с `积分 >= 2` обработаны (установлено `已下沉 = 是`)
- Актуальная проблема блокирующего характера выявлена и предотвращена
- Программа phòng ngừa потока обновлена / исправлена
- Успешный навык создан / улучшен
- Чистота слоев (L2/L3/Skill) подтверждена

## Типичные ошибки
- ❌ **Пропустить карточку интеграции**: Некоторые категории недостаточно интегрированы
- ❌ **Блокирующее**: Отсутствие доказательной причины (напр., 「просто не заметил»)
- ❌ **Не обновлять L1**: Доверять, что проблема исчезнет сама (не)
```

## 5.2 Конфигурация на своей платформе

### 5.2.1 Варианты внедрения

| Вариант | Как внедрить | Уровень внедрения | Requirements |
| :--- | :--- | :--- | :--- |
| **External Skill Prompt** | Загрузить навыки как встроенные prompt | Low | Поддержка загрузки prompt |
| **Built-in L1** | Встроить L1 в системный prompt | Medium | Доступ к редактированию system prompt |
| **Plugin** | Develop a plugin implementing the whole arch | High | Поддержка extension/plugin система |

**Рекомендация:** начать с External Skill Prompt → когда стабильно, рассовать Built-in.

### 5.2.2 External Skill Prompt (External 预馈)

**Плюсы и минусы:**

- ✓ Быстро реализовать (нужен только prompt loading)
- ✓ Не влияет на системный prompt
- ✓ Может быть быстро отключен/изменен для experiment
- ✗ Каждый разtask loading нужно явно вызывать
- ✗ Зависит от того, как platform реализует prompt loading

**Implementation:**

1. Создать три файла: `cybernetics-feedforward-startup.md`, `cybernetics-delivery-gate.md`, `cybernetics-control-loop-review.md`
2. На вашей платформе найти функцию загрузки навыков / prompt
3. Загрузить эти навыки
4. Указать логике агента: «каждая задача начало → вызите `cybernetics-feedforward-startup` », «перед выводом → вызите `cybernetics-delivery-gate` », «после фазы → вызите `cybernetics-control-loop-review` ».

### 5.2.3 Built-in L1 (Встроенный system.prompt)

**Плюсы и минусы:**

- ✓ Не нужно каждый раз явно вызывать
- ✓ Атоматически применяется ко всем задачам
- ✗ Изменение 为 сложного системного настроения

**Implementation:**

1. Найти system prompt configuration (обычно в config / platform settings)
2. Добавить L1 правила (в AS-IS формате из `cybernetics-feedforward-startup.md`)
3. ПодключитьL1 retrigger mechanism (retrigger 下层workflow)
4. Испытание / лучшие практики с первого небольшого project

### 5.2.4 Plugin-based (Плагинного внедрения)

Если ваша платформа поддерживает plugins, реализуйте шарим к размеру для внедрения всей архитектуры. Это даёт наилучшую flexibility, но требует больше работы.

## 5.3 Проверка и тестирование

### 5.3.1 Unit Tests саманом integration

**Test Scenario 1: Feedforward Startup**

1. Создать новую запись памяти: «Пользователь любит короткие ответы»
2. Запустить feedforward startup (вызывать `cybernetics-feedforward-startup`)
3. Verify:
   - Предпочтение загружено в контекст
   - Контекст содержит слово 「короткий」

**Test Scenario 2: Integral Control / Points Table**

1. Пользователь: «Ты слишком пространно»
2. Проверить:
   - Создана запись в `积分控制表`:`沟通 | 太空间но | <today> | 1 | 2 | 否 | L2沟通原则`
3. Повтор反馈:
4. Проверить:
   - Запись обновлена:`累计次数 = 2`
   - Создана запись памяти: 「L2沟通原则: 避免多余的 или пространным высказываниям, держите ответ кратким」
   - Запись обновлена:`已下沉 = 是`

**Test Scenario 3: Delivery Gate**

1. Создать файл с 0 байтами (п intentionally)
2. Пытаться выполнить `write_file(path='zero-byte.txt', content='')`
3. Verify:
   - Delivery gate detcoeffected: tiêu раздела P1 — файл操作
   - P1 FAILED
   - Agent не выводит
   - Agent исправляет (rewrite with correct content)
   - Re-run delivery gate: P1 PASSED

### 5.3.2 集成 verification

**Scenario:** Выполнение всей цепочки workflow

1. Пользователь: «Напиши unit test для этой функции»
2. Agent:
   - Trigger feedforward startup
     - Loading user preference: «Пользователь предпочитает code-first testing»
     - Loading L3 pattern: «测试前调用 skill 进行.systematic testing»
   - Execution:
     - Write test
     - Результаты тестов: pass
   - Trigger delivery gate:
     - P0: syntax OK
     - P0: exit code 0
     - P1: файл操作 безопасно (write→validate→удаление)
     - PASSED
   - Вывод

3. Verify:
   - Loaded user preference и context references
   - Test выполнен и прошел
   - Delivery gate использовался и проверил файл操作
   - Output без P0/P1 ошибки

### 5.3.3 常见问题

**问题:** My platform не поддерживает отдельное skill loading.
**答:** Режим «memory simulation»: запишите весь skill content как запись памяти с типом 「skill」. В L1, перед началом tasks, автоматическиretrieval и execute records типа 「skill」 (не query, а read records). 缺点 требует больше token em для памяти.

**问题:** feedforward startup занимает много времени.
**答:** Оптимизируйте через фильтрацию:
- Только Направить китайским keyword (prefers, 喜, 佰, 不喜)
- Ограничить результат до latest 3-5 records
- Если записей страниц много, запросить пользователя и индексироваться вручную, а не напряглайнепрограмма каждый раз.

**问题:** points control table 如何更新?
**答:** Атоматически更新 в каждый раз получения негативного обратного сигнала:
1. Detect negative feedback
2. Extract error pattern
3. Add/update record in points control table
4. Triggered processes в feedforward startup

## 5.4 Продвинутая настройка (по желанию)

### 5.4.1 Customizing threshold для разных categories

По умолчанию threshold=2 для всех categories. Для некоторых sensitive categories (например, safety/security), можно установить threshold=1.

Пример реализации:

```
修改 `积分控制表` 每行新增字段 `自定义阈值`:

类别 | 错误模式(≤8字) | 日期 | 累计次数 | 阈值(2) | 自定义阈值 | 已下沉 | 下沉目标
```

Интегральный control iteration:

```
if 自定义阈值 存在:
    threshold = 自定义阈值
else:
    threshold = 2

if 累计次数 >= threshold:
    # Trigger sink
```

### 5.4.2 Расширение delivery gate с自定义 проверками

Add custom P0/P1/P2检查 к delivery gate for specific concerns.

Example: для Prometheus周报生成る задач custom check 检查 ошибок:

```
Add to `cybernetics-delivery-gate.md`:

P1 (Prometheus周报特有):
  [ ] No "N/A" placeholders in report
  [ ] At least 10 metrics are analysed
  [ ] ROI is calculated and included
```

### 5.4.3 Уровневая alignment для personal assistant ширины

Если используется multiple agents分工 (например, 监控agent + 开发agent + 写текстыagent), alignment l1/l2/l3/skill каждому agent:

- L1: Волшебных константы единые между agents
- L2: Agent-specificдля各有不同的角色
- L3: Shared reusable patterns (например, 测试workflow、окружение检查)
- Skills: Shared core skills + agent-specific skills
