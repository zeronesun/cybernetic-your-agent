# Глава 3: Четыре концепции кибернетики — подробный разбор

Эта глава глубоко объясняет четыре concepts, извлечённые из инженерной кибернетики Цянь Сюэсэнь, и демонстрирует, как их использовать в практических сценариях AI-агентов.

## 3.1 Предварительная связь — избегание проблем до того, как они произойдут

### 3.1.1 ConceptCore

Предварительная связь — это операция упреждения. В системах управления, прежде чем система отклонится от цели, контроллер активно загружает внешние disturbы и сценарии и делает提前 членеции. В системе агентов, мы меняем это в: **перед началом задачи, активно загружать пользовательские предпочтения, исторические уроки и паттерны известной ловушки, чтобы подготовиться.**

### 3.1.2 Почему это нужно для AI-агентов

В следующей ситуации часто можно наблюдать:

Пользователь впервые говорит агенту: «Я люблю короткие, проницательные ответы. Пожалуйста, не начинайте с «Я понимаю...»Agent: «Хорошо, я запомнил это.»Через неделю та же просьба: «Пожалуйста, не начинайте с «Я понимаю...»Agent: «Хорошо, я запомню это.»Представляет ли он «Я понимаю» каждую секунда? 95% — да. Вопрос не в том, что agent не понимает, а в том, что он **забыл его.**

Системы памяти большинства агентов не имеют механизма «предварительной загрузки». Они пассивно реагируют на внутренний запрос (например, ввод пользователя), а затем ищут память. Если память — это «словарь», без предварительной загрузки, она даже не знает, когда проверять.

### 3.1.3 Как реализовать

В уровне L1 метакогнитивного фреймворка добавьте принцип работника:

**Правило**: Task Startup → Всегда вызывайте предварительную связь механизма
**Shenshu**: Перед началом любого задачи,Agent должен сделать три вещи:

1. Trigger предварительной связи:**Проверьте пользовательские предпочтения в долгосрочной памяти**
   - Search: `prefers`、`喜欢`、`prefers`、`ценивает`、`не喜`、`不喜`、` prefers short`、`does not like verbose`
   - Load все соответствующие записи (带时间戳的)
   - Сортируйте по времени, берёте последние 3-5 домашних
   - После сортировки вставьте в текущий контекст

2. Загрузить исторические классы:**Проверьте предыдущие параграф. в длинной таблице контроля ошибок**
   - Search: `积分控制表`、`积分控制`、`积分`
   - Load сами «积分控制表» (если существует)
   - Filter: берёте только `积分计数≥阈值(2)` 的记录
   - Insert в контекст как **警告**: «[积分控制] 该模式已触发≥2次, 必须遵循该规则.»

3. Загрузить паттерны сцены:**Проверьте L3Base Patterns**
   - Search: `L3`、`ядрное паттерн`、`pattern`
   - Load паттерны для этой типа задач
   - Example: для 「写故事」 задачи, load `L3: 核心pattern — 情感基调优先 情带宽, 留白在结尾`

После трёх шагов, контекст теперь содержит: пользовательские предпочтения + шансы обучения + паттерны сцены.

Только после этого можно начать реальное думать и действовать.

### 3.1.4 Пример практический

**Before (нет предварительной связи):**

```
Agent: (нет контекста пользовательских предпочтений)
"我理解您的意思..." (trigger用户不喜)
```

**After (с предварительной связью):**

```
Agent: (trigger预馈, 自动load: 「不喜 verbose 开头, 喜欢short直接」)
"您的需求是..."
```

### 3.1.5 Частые проблемы

**问**: 记忆条目很多时, 怎么定优先加载哪些?
**答**: 带 timestamp, 优先最新的3-5条. 用户的偏好会演变, 越新的越能代表当前偏好.

**问**: 如何记用户偏好?
**答**: 人工维护, 每10次合作中有1-2次, 主动问用户 「我最近的表达您觉得如何? 有需调整吗?」 并将反馈写入 `user` 类型的记忆.

**问**: 预馈需要数据库吗?
**答**: 不需要, 依赖平台的记忆功能反查; 平台不支持反查时, 人工维护 `feedforward.md` 文件存要做预馈的内容.

## 3.2 积分控制 — 防振动的稳定反馈阈值

### 3.2.1 核心概念

积分控制是从控制器理论中借来的概念: 积分器只累积偏差, 当超过阈值才采取行动.

简单说: **一次投诉是噪声, 两次才是信号.**

在Agent系统中实现为:

- 首次负面反馈: 临时修正 (不写入长期记忆/技能)
- 再次同类负面反馈: 写入记忆/技能, 形成长期约束
  - 记录到 `积分控制表`
  - 计数递增 (`积分 += 1`)
  - 当 `积分 ≥ 2` 时, 触发「写入记忆/技能」

### 3.2.2 为什么需要

没有积分阈值时, Agent:

- 用户一吐槽就改所有事情 (overreaction / 振荡)
- 忽略掉真正稳定的模式 (因为都在动态)

积分控制提供「噪声过滤 + 模式提取」:

- 过滤掉单次偶发的抱怨不是偏好
- 提取出重复出现的偏好才是真偏好

### 3.2.3 如何实现

**机制设计:**

建立 `积分控制表`, 每条记录包含:

| 字段 | 说明 | 示例 |
| :--- | :--- | :--- |
| 类别 | 错误/偏好归属的领域 | `沟通`、`决策`、`文风` |
| 错误模式 | 简短的模式描述 (≤8字符) | `太啰嗦`、`删除后写` |
| 日期 | 记录时间 | `2025-01-15` |
| 累计次数 | 负面反馈出现次数 | `2` |
| 阈值 | 触发下沉目标 | `2` |
| 已下沉 | 是否已化入L2/L3 | `是` / `否` |
| 下沉目标 | 沉入哪个层级 | `L2沟通原则`、`L3文风pattern` |

**触发流程:**

1. **收到负面反馈**

   Agent 检测到用户明确负面信号:
   - 「不用」 (只说这句话)
   - 「你说的不对」
   - 「我上次跟你讲了xxx怎么又这样了」
   - 「我不喜欢这种表达」
   - 非请求性提出的「错了」、「不对」

2. **提取错误模式**

   - 将负面信息压缩为≤8字符模式
   - Example:
     - 「你太啰嗦了, 不要over explaining」 → `太啰嗦`
     - 「删除后写很危险, 应该先写后删」 → `删除后写`

3. **检查积分控制表**

   - 如果该模式不存在: 创建新记录, `累计次数 = 1`, `已下沉 = 否`
   - 如果该模式存在: `累计次数 += 1`

4. **判断触发**

   ```
   if 累计次数 >= 阈值(2) and 已下沉 == 否:
       # 触发「下沉」
       
       # 1. 写入长期记忆/技能
       if 下沉目标 /starts_with/ "L2":
           写入 `memory`, target = "memory"
       else:
           创建/更新 skill, 名字 = 下沉目标
       
       # 2. 更新积分控制表
       已下沉 = 是
   ```

5. **更新积分控制表**

   写入回 `积分控制表` 条目.

### 3.2.4 实际例子

**首次反馈:**

```
用户: 「你太啰嗦了, 不要over explaining」
Agent: 提取模式=`太啰嗦`
        创建记录: 沟通 | 太啰嗦 | 2025-01-15 | 1 | 2 | 否 | L2沟通原则
```

**二次反馈:**

```
用户: 「上次又太啰嗦了, 记住了没?」
Agent: 提取模式=`太啰嗦`
        查找记录: 已存在, count=1 → count+=1 → 2
        判断: count≥2 且 未下沉 → 触发写入L2
        写入 memory: 「L2沟通原则: 严禁啰嗦表述, 保持短直接」
        更新记录: 已下沉=是
        输出: 「已记入长期原则」
```

### 3.2.5 术语表: 各种模式的下沉目标

下表为常见错误模式与对应下沉目标的对照表, 规范化以免重复创建 / 名称不一致.

| 错误模式 | 下沉目标 | 说明 |
| :--- | :--- | :--- |
| `太啰嗦` | `L2沟通原则` | 沉入L2用户底层逻辑, 每次回复都检查 |
| `删除后写` | `L2决策原则` | 沉入L2, 文件操作必须 写→验→删, 严禁先删后写 |
| `任务理解不准` | `L3任务pattern` | 沉入L3, 该类型任务启动时优先确认理解 |
| `测试未全` | `skill` (特化技能) | 沉入技能, 调用系统化测试ops |
| `函数选错` | `skill` | 沉入技能, 该领域有专有函数 |
| `数据去噪` | `skill` | 沉入技能, 某任务先做数据清洗 |

**第一列的「错误模式」必须 ≤8 字符** (为了方便识别和匹配); 超出的, 应当做进一步的语义压缩, 举例: `'先别管git, 我自己来'` 可压缩为 `'阻止git自动化'`.

### 3.2.6 常见问题

**问**: 负面反馈如何 自动 检测?
**答**:
1. 负向词: `不用`、`不对`、`错了`、`错误的`、`not correct`、`wrong`
2. 不夹杂请求的单独负面: 「不用」(无后续动作)、「你说的不对」(无请求)
3. 「我上次讲了xxx怎么又这样了」 表明重复错误
4. 「我不喜欢/偏好的」 明确表达偏好

凡是满足以上任一, 则归类为「负面反馈」.

**问**: 首次反馈是否要做任何动作?
**答**: 临时修正 (回答问题本身 / 丹()) 但不做任何长期沉淀.

**问**: 阈值为何设为2?
**答**: 一次可能是噪声/偶发抱怨, 两次是稳定偏好; 更低阈值会产生振荡, 更高阈值会让真模式被过滤掉; 该值已通过大量实验优化.

**问**: 如何维护/访问 `积分控制表`?
**答**:
- 作为 `memory` 类型为 `积分控制表` 的条目存在
- Advisor维护: 每次 negative feedback 后更新
- Agent访问: 每次任务启动前, 读取 `积分控制表`, 筛选 `积分 ≥ 2` 的记录, 作为警告插入上下文

## 3.3 层级分离 — 防止混淆, 提升信噪比

### 3.3.1 核心概念

层级分离是将信息和 instruction 分离到不同「层」:

- **L2: 用户底层逻辑 (Why)** — 只存「为什么」(原则、偏好)
- **L3: 场景核心pattern (What)** — 只存「该类任务该遵循什么模式」
- **Skill: 执行层 (How)** — 只存「具体步骤、checklist、API、坑点」

**严格禁止**: 技能里混入「为什么」(历史背景、冗长说明), 或记忆里混入「详细步骤」(长instruction).

**分层的好处**:

1. **提升信噪比**: 记忆只存高价值原则; 混入详细操作程序会 decadence记忆, 造成信息污染.
2. **防模板化**: 「为什么」是有个性的(用户L2), 「怎么做」是可模板化的(skill). 技能可以跨用户、跨平台复用; 带入用户个性化原则会破坏这个性质.
3. **性能优化**: L1加载时检索目标明确(查L2原则、L3pattern); 若什么都塞进记忆, 则检索无目标, 消耗token.

### 3.3.2 典型错误示例

**错误1: 技能里塞历史背景**

```markdown
## SKILL.md
历史: 2024年5月, 用户反馈任务理解经常偏...
步骤: 1. 确认任务 2. 计划...
```

问题: 历史背景是「为什么」(L2), 不应在技能(skill)中出现.

**修正**:

```markdown
# L2 (在 memory 里)
「L2·任务理解 — 用户偏好: 凡是涉及环境/项目操作前, 必须pwd+ls+项目识别, 不猜测用户意图.」

# SKILL.md (在技能里)
## 02-确认任务模式
1. pwd
2. ls -la
3. 问: "是什么项目?"
```

**错误2: 记忆里塞详细步骤**

```markdown
记忆条目:
「步骤: 1. 安装; 2. 配置 ALLOWED_HOSTS; 3. 配置 Vite...」
```

问题: 详细步骤是「怎么做」, 应在技能文件, 不应在记忆条目.

**修正**:

```markdown
# 在 memory 里
「L2技术栈偏好: Django+Vue全栈, 前端3000/后端8000, ALLOWED_HOSTS含 WSL IP」

# SKILL.md
详细步骤第N步: 配置ALLOWED_HOSTS...
```

### 3.3.3 检验清单

创建/更新记忆/技能时, 自检:

- **如果是记忆项**:
  - [ ] 是否包含「为什么」/「原则」/「偏好」? ✓ (应该)
  - [ ] 是否包含>3行的详细步骤? ✗ (不应)
  - [ ] 是否包含API调用/具体命令? ✗ (不应)

- **如果是技能文件**:
  - [ ] 是否包含历史背景/用户故事? ✗ (不应)
  - [ ] 是否包含详细步骤/清单? ✓ (应该)
  - [ ] 是否包含坑点/API用法? ✓ (应该)

### 3.3.4 常见问题

**问**: L2和L3的区别是什么?
**答**:
- **L2 (用户底层逻辑)** — 跨场景的通用原则
  - Example: 「凡事代码优先分析」、「不猜测用户意图直接确认」
  - 适用: 「沟通」、「决策」等

- **L3 (场景核心pattern)** — 特定任务类型的核心pattern
  - Example: 「故事创作 — 情感基调优先, 留白在结尾」
  - 适用: 「写故事」、「Django架构」等

**问**: 什么时候下沉到L2 vs L3?
**答**:
- 如果错误模式跨场景存在 (如「沟通太啰嗦」), 下沉到L2
- 如果错误模式特定某类任务 (如「测试未全」), 下沉到L3

**问**: 如何检查现有记忆/技能是否符合层级分离?
**答**: 运行以下脚本, 自动识别违反分离原则的条目 (skill里出现「历史/用户故事」、记忆里出现「步骤»等).

## 3.4 交付自检 — 不合格不交付

### 3.4.1 核心概念

在输出任何结果前, 穿过「交付自检闸门」, 执行三级风险分级检查 (P0/P1/P2/P3).

```
任务完成 → 交付自检闸门 → (全过) → 输出
               ↓
            (任一不过) → 不输出 → 修复/重新做 → 重新检查
```

**仅当所有检查全部通过才交付**.

### 3.4.2 风险分级

| 级别 | 定义 | 检查内容 | 示例 |
| :--- | :--- | :--- | :--- |
| **P0** | 致命错误, 立即返回 | 语法错误、运行时异常、命令返回非0退出码 | `error: SyntaxError` |
| **P1** | 功能缺陷, 导致结果不可用 | 数据丢失、文件操作反序 (先删后写)、测试未全 | 新文件0字节 |
| **P2** | 体验/质量噪声, 需要纠正 | 啰嗦冗余、格式不一致、术语混用 | 「不再赘述」 |
| **P3** | 建议/优化机会 | 优化点、可读性 | 变量名得更长 |

**规则**:

- P0/P1 **必须通过** → 否则不输出
- P2 **建议纠正** → 纠正后再输出
- P3 **可选** → 输出时附带说明建议

### 3.4.3 闸门实现

**在 L1 metacognitive framework 定义:**

```
规则: 任一输出前 → 必须执行交付自检闸门

检查项:

P0级检查:
  [ ] 语法: Python/Shell语法是否合法
  [ ] 运行时: 命令执行是否返回 error
      (error在 tool_output 的 error 字段 或 exit_code != 0)
  [ ] 类型匹配: 文件操作使用了文件列表当做字符串?

P1级检查:
  [ ] 文件操作: 是否使用了「先删后写」的安全顺序?
      (应: 写→验→删)
  [ ] 数据完整性: 测试是否覆盖所有关键场景?
  [ ] 权限: 操作是否尝试sudo? (禁止)

P2级检查:
  [ ] 啰嗦冗余: 回复是否超过50%是重复模式语句/无意义套话?
  [ ] 格式一致性: 同一术语多次使用是否混用中英文?
  [ ] 术语一致性: feedforward/피드포워드/前饋 是否混用?

P3级检查:
  [ ] 可读性: 变量名/函数名是否足够长说明作用?
  [ ] 性能: 是否有重复计算可缓存?

触发: 任一P0/P1不过 → 不输出; P2需纠正; P3建议
```

### 3.4.4 实际例子 (FAILED后如何修复)

**场景:** 写入新文件后, 没有验证文件存在就删除旧文件

**闸门检测:**

```
[P1] 文件操作: 检测到「先删后写」危险顺序 → FAILED
```

**修复:**

1. 不输出
2. 重新编排操作: 写→验→删
3. 重新执行
4. 重新检查

**修复后输出:**

```
✓ 交付自检闸门 — 全过

完成...
```

### 3.4.5 常见问题

**问**: P2检查不通过会怎么样?
**答**: 会尝试自动纠正 (如去除冗余、统一术语). 纠正后再次检查, 直到通过或到最大尝试次数.

**问**: P3是可选的, 为什么还要检查?
**答**: P3是通过后的建议, 输出时可附带「优化建议: ...」以提升 code 质量.

**问**: 如何添加新的检查项?
**答**: 在 L1 metacognitive framework 中扩展 `delivery-gate` 的检查清单; 该清单可动态增减检查点.

## 3.5 四个概念的小结

| 概念 | 作用 | 如何调用 |
| :--- | :--- | :--- |
| **预馈** | 任务启动前主动加载偏好、教训、pattern | Task startup → 必 trigger 预馈机制 |
| **积分控制** | 1次投诉是临时, 2次才系统改 | 收到负面反馈 → 提取模式 → 更新积分表 → 判断触发 |
| **层级分离** | 记忆存「为什么」, 技能存「怎么做」 | 创建/更新记忆/技能时, 自检是否违反分离原则 |
| **交付自检** | 输出前自动P0-P2分级检查不劳则不交付 | 任一输出前 → 必走交付自检闸门 |

接下来, 第四章将揭开完整架构: 三层记忆体 → 技能层 → 数据流.
